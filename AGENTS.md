# 智能体调用关系图

本文档定义了各智能体之间的调用链和协作关系，确保任务能够高效、有序地完成。

## 智能体分类

### 🎯 协调层（Coordination Layer）

| 智能体       | 职责       | 调用时机               |
| ------------ | ---------- | ---------------------- |
| orchestrator | 中央协调器 | 复杂任务、多智能体协作 |
| planner      | 实施规划   | 功能实现、架构变更     |

### 🏗️ 架构层（Architecture Layer）

| 智能体    | 职责            | 调用时机             |
| --------- | --------------- | -------------------- |
| architect | 系统架构+云架构 | 新功能设计、架构决策 |

### 🔍 审查层（Review Layer）

| 智能体            | 职责           | 调用时机                 |
| ----------------- | -------------- | ------------------------ |
| code-reviewer     | 多语言代码审查 | 所有代码变更             |
| security-reviewer | 安全审查       | 用户输入、认证、敏感数据 |
| database-reviewer | 数据库审查     | SQL、迁移、模式设计      |

### 🔧 修复层（Fix Layer）

| 智能体           | 职责           | 调用时机           |
| ---------------- | -------------- | ------------------ |
| build-resolver   | 多语言构建修复 | 构建失败           |
| refactor-cleaner | 代码清理       | 死代码、重复项清理 |

### 🧪 测试层（Testing Layer）

| 智能体 | 职责             | 调用时机              |
| ------ | ---------------- | --------------------- |
| qa     | 测试策略+TDD+E2E | 新功能、Bug修复、重构 |

### 🚀 部署层（Deployment Layer）

| 智能体 | 职责            | 调用时机           |
| ------ | --------------- | ------------------ |
| devops | CI/CD+Git工作流 | 部署配置、基础设施 |

### 🎨 专业层（Specialist Layer）

| 智能体                | 职责     | 调用时机           |
| --------------------- | -------- | ------------------ |
| performance-optimizer | 性能优化 | 性能瓶颈分析       |
| fullstack             | 全栈开发 | ML、移动、数据、UX |

### 📚 支持层（Support Layer）

| 智能体      | 职责     | 调用时机              |
| ----------- | -------- | --------------------- |
| doc-updater | 文档更新 | 代码映射、README 更新 |

## 协作规则

### 优先级规则

```
安全 > 功能 > 性能 > 样式
```

当多个智能体提供冲突建议时：

1. **安全相关** — 最高优先级，必须解决
2. **功能相关** — 次高优先级，影响用户
3. **性能相关** — 中等优先级，可后续优化
4. **样式相关** — 最低优先级，可选改进

| 语言                  | 代码审查      | 构建修复       | 测试框架     |
| --------------------- | ------------- | -------------- | ------------ |

| Python                | code-reviewer | build-resolver | pytest       |
| Go                    | code-reviewer | build-resolver | go test      |
| Rust                  | code-reviewer | build-resolver | cargo test   |
| Swift                 | code-reviewer | build-resolver | XCTest       |
| Java                  | code-reviewer | build-resolver | JUnit        |
| Kotlin                | code-reviewer | build-resolver | JUnit        |

## 后续委托规则

每个智能体完成任务后，应根据情况委托给下一个智能体：

### 代码审查类

| 完成后        | 委托条件       | 委托目标          |
| ------------- | -------------- | ----------------- |
| code-reviewer | 发现安全问题   | security-reviewer |
| code-reviewer | 发现数据库问题 | database-reviewer |
| code-reviewer | 发现构建错误   | build-resolver    |
| code-reviewer | 需要重构       | refactor-cleaner  |

### 开发类

| 完成后    | 委托条件 | 委托目标      |
| --------- | -------- | ------------- |
| architect | 需要实施 | planner → qa  |
| planner   | 需要开发 | qa            |
| qa        | 代码完成 | code-reviewer |

### 测试类

| 完成后 | 委托条件 | 委托目标      |
| ------ | -------- | ------------- |
| qa     | 需要修复 | qa (TDD)      |
| qa     | 测试失败 | code-reviewer |

### 部署类

| 完成后 | 委托条件     | 委托目标          |
| ------ | ------------ | ----------------- |
| devops | 需要安全审查 | security-reviewer |
| devops | 需要文档     | doc-updater       |

### 专业类

| 完成后                | 委托条件   | 委托目标              |
| --------------------- | ---------- | --------------------- |
| performance-optimizer | 代码优化   | code-reviewer         |
| performance-optimizer | 数据库优化 | database-reviewer     |
| performance-optimizer | 架构调整   | architect             |
| fullstack             | API 集成   | code-reviewer         |
| fullstack             | 部署配置   | devops                |
| fullstack             | 安全审查   | security-reviewer     |
| fullstack             | 性能优化   | performance-optimizer |

### 版本控制类

| 完成后        | 委托条件         | 委托目标      |
| ------------- | ---------------- | ------------- |
| devops        | 代码需要审查     | code-reviewer |
| devops        | 合并后需要测试   | qa            |
| devops        | 发布需要部署     | devops        |
| devops        | 需要更新文档     | doc-updater   |
| code-reviewer | 审查通过准备合并 | devops        |

## 最佳实践

### 1. 避免循环调用

```
❌ 错误：A → B → A
✅ 正确：A → B → C → 完成
```

### 2. 保持调用链简洁

```
❌ 错误：orchestrator → planner → architect → planner → qa
✅ 正确：orchestrator → planner → architect → qa
```

### 3. 条件判断前置

在调用前判断是否需要调用下一个智能体，避免不必要的调用。

### 4. 上下文传递

在智能体之间传递必要的上下文信息，避免重复分析。

### 5. 错误处理

当某个智能体无法完成任务时，应回退到上一个智能体或报告给 orchestrator。
