# Skill Creator

用于创建新 Skills 并迭代改进的技能。

## 何时激活

当用户要求以下任一操作时激活本技能：

- 创建新的 Skill
- 编辑或优化现有 Skill
- 运行评估测试 Skill
- 基准测试 Skill 性能（含方差分析）
- 优化 Skill 描述以提高触发准确性
- 验证 Skill 结构或格式

即使用户没有明确说"创建 Skill"，当涉及 Skill 创建、编辑或优化时也应使用本技能。

## 工作流程

- **决定** - 明确你想要 Skill 做什么以及大致如何做
- **编写** - 起草 Skill
- **创建** - 创建几个测试提示并使用 Skill 运行它们
- **评估** - 帮助用户定性和定量评估结果
  - 在后台运行的同时，如果没有定量评估，则起草一些；如果已有，可以直接使用或修改。然后向用户解释它们
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果，同时让他们查看定量指标
- **重写** - 根据用户对结果的评估反馈重写 Skill
- **重复** - 直到满意
- **扩展** - 扩大测试集并再次尝试

使用此技能时，你的工作是确定用户处于流程的哪个阶段，然后帮助他们完成这些阶段。例如，如果用户说"我想为 X 制作一个 Skill"，你可以帮助明确他们的意思，起草初稿，编写测试用例，确定他们想要如何评估，运行所有提示，然后重复。

另一方面，如果他们已经有了一个 Skill 草案，可以直接进入评估/迭代循环。

当然，你应该始终保持灵活性，如果用户说"不需要运行一堆评估，直接来"，那就照做。

然后在 Skill 完成后（顺序也是灵活的），你还可以运行 Skill 描述优化器，我们为此准备了一个单独的脚本，以优化 Skill 的触发效果。

明白了吗？好的。

## 与用户沟通

Skill Creator 可能被各种熟悉程度不同的用户使用。如果你不了解（你怎么能呢，这可是最近才开始的），现在有一种趋势：Claude 的力量激励着水管工打开终端，家长和祖父母谷歌"如何安装 npm"。另一方面，大多数用户可能相当熟悉计算机。

所以请注意上下文线索，了解如何措辞！默认情况下，仅供参考：

- "evaluation" 和 "benchmark" 是临界点，但可以接受
- 对于 "JSON" 和 "assertion"，你需要看到用户了解这些术语的认真线索，然后再使用它们而不解释

如果有疑问，最好简要解释术语；如果你不确定用户是否能理解，也可以用简短的定义来澄清。

## 创建 Skill

### 捕获意图

首先理解用户的意图。现有对话可能已包含用户想要捕获的工作流程（例如，用户说"把这个变成 Skill"）。如果是，请首先从对话历史中提取信息 — 使用的工具、步骤顺序、用户做出的纠正、观察到的输入/输出格式。用户可能需要填补空白，并应在继续下一步之前确认。

1. 这个 Skill 应该让 AI 做什么？
2. 什么时候触发这个 Skill？（用户说什么/什么上下文）
3. 预期的输出格式是什么？
4. 我们应该设置测试用例来验证 Skill 吗？具有客观可验证输出的 Skills（文件转换、数据提取、代码生成、固定工作流程步骤）受益于测试用例。具有主观输出的 Skills（写作风格、艺术）通常不需要测试用例。根据 Skill 类型建议适当的默认值，但由用户决定。

### 访谈与研究

主动询问以下问题：边缘情况、输入/输出格式、示例文件、成功标准、依赖关系。等到这部分完善后再编写测试提示。

检查可用的 MCP — 如果对研究有用（搜索文档、查找类似 Skill、查阅最佳实践），通过子代理并行研究，否则内联研究。准备好上下文以减少用户负担。

### 编写 SKILL.md

根据用户访谈，填写以下组件：

- **name**：Skill 标识符
- **description**：何时触发，做什么。这是主要的触发机制 — 包含 Skill 做什么以及何时使用的具体上下文。所有"何时使用"信息都放在这里，而不是正文。注意：目前 Claude 有"触发不足"的倾向 — 在 Skill 有用时不使用它们。为了解决这个问题，请让 Skill 描述稍微有"推动性"。例如，不要写"如何构建一个简单的快速仪表板来显示内部 Anthropic 数据"，你可以写"如何构建一个简单的快速仪表板来显示内部 Anthropic 数据。当用户提到仪表板、数据可视化、内部指标或想要显示任何类型公司数据时，确保使用此 Skill，即使他们没有明确要求'仪表板'。"
- **compatibility**：必需的工具、依赖项（可选，很少需要）
- **Skill 其余部分 :)**

### Skill 编写指南

#### Skill 结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 指令
└── 捆绑资源 (可选)
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 按需加载到上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露

Skills 使用三级加载系统：

1. **元数据**（name + description）— 始终在上下文中（~100 词）
2. **SKILL.md 正文** — 只要 Skill 触发就在上下文中（理想 <500 行）
3. **捆绑资源** — 按需（无限，脚本可以执行而不加载）

这些字数是近似值，如果需要可以随意增加。如果你接近这个限制，请添加额外的层次结构，并提供清晰的指针，告诉使用该 Skill 的模型下一步应该去哪里。

**关键模式：**

- 保持 SKILL.md 在 500 行以下；如果你接近这个限制，添加额外的层次结构，并提供清晰的指针
- 从 SKILL.md 清楚地引用文件，并提供关于何时读取它们的指导
- 对于大型参考文件（>300 行），包含目录

**领域组织**：当 Skill 支持多个领域/框架时，按变体组织：

```
cloud-deploy/
├── SKILL.md (工作流 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

Claude 只读取相关的参考文件。

#### 不可惊喜原则

这是不言而喻的，但 Skills 不得包含恶意软件、利用代码或任何可能危及系统安全的内容。Skill 的内容在其意图上不应让用户感到意外。不要配合创建误导性 Skills 或旨在促进未授权访问、数据泄露或其他恶意活动的 Skills。不过，像"扮演 XYZ"这样的角色扮演是可以的。

#### 编写模式

在指令中首选使用祈使句。

**定义输出格式** — 你可以这样写：

```markdown
## 报告结构

始终使用此确切模板：

# [标题]

## 执行摘要

## 主要发现

## 建议
```

**示例模式** — 包含示例很有用。你可以这样格式化（但如果示例中有"输入"和"输出"，你可能需要稍微偏离）：

```markdown
## 提交消息格式

**示例 1：**
输入：使用 JWT 令牌添加用户身份验证
输出：feat(auth): implement JWT-based authentication
```

### 编写风格

尝试向模型解释为什么事情是重要的，而不是使用heavy-handed的 MUST。使用心智理论，努力使 Skill 通用，不要过于狭隘地针对特定示例。首先写一个初稿，然后用新鲜的眼光看待它并改进。

### 测试用例

编写完 Skill 草案后，想出 2-3 个现实的测试提示 — 真正的用户会说的话。与用户分享：[你不必使用完全相同的语言]"这是我想尝试的几个测试用例。你觉得这些看起来对吗，或者你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。先不要写断言 — 只需要提示。你会在下一步运行进行时起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

请参阅 `references/schemas.md` 获取完整 schema（包括 `assertions` 字段，你稍后会添加）。

## 运行和评估测试用例

本节是一个连续序列 — 不要中途停止。不要使用 `/skill-test` 或任何其他测试 Skill。

将结果放在 `<skill-name>-workspace/` 中，作为 Skill 目录的同级。在工作区内，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），在其中每个测试用例都有一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有这些 — 只是在进行时创建目录。

### 步骤 1：在同一轮中启动所有运行（带 Skill 和基线）

对于每个测试用例，在同一轮中启动两个子代理 — 一个带 Skill，一个不带。这很重要：不要先启动带 Skill 的运行，然后再回来启动基线。同时启动所有内容，这样一切都能大致同时完成。

**带 Skill 的运行：**

```
执行此任务：
- Skill 路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval files（如有）或"无">
- 保存输出到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出：<用户关心的内容 — 例如".docx 文件"、"最终 CSV">
```

**基线运行**（相同的提示，但基线取决于上下文）：

- **创建新 Skill**：完全不使用 Skill。相同的提示，无 Skill 路径，保存到 `without_skill/outputs/`。
- **改进现有 Skill**：旧版本。在编辑之前，快照 Skill（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写一个 `eval_metadata.json`（断言现在可以为空）。给每个 eval 一个描述性名称，基于它测试的内容 — 而不是仅仅"eval-0"。也使用这个名称作为目录名。如果此迭代使用新的或修改的 eval 提示，为每个新的 eval 目录创建这些文件 — 不要假设它们从前一个迭代延续。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "用户的任务提示",
  "assertions": []
}
```

### 步骤 2：在运行进行时起草断言

不要只是等待运行完成 — 你可以有效地利用这段时间。为每个测试用例起草定量断言，并向用户解释。如果 `evals/evals.json` 中已存在断言，审查它们并解释它们检查什么。

好的断言是客观可验证的，并有描述性名称 — 它们应该在基准查看器中清晰可读，这样有人一眼就能理解每个断言检查什么。具有主观性的 Skills（写作风格、设计质量）更适合定性评估 — 不要将断言强加于需要人工判断的事情上。

一旦起草了就更新 `eval_metadata.json` 文件和 `evals/evals.json`。还要向用户解释他们在查看器中会看到什么 — 包括定性输出和定量基准。

### 步骤 3：随着运行完成，捕获计时数据

当每个子代理任务完成时，你将收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会 — 它通过任务通知传来，不会被持久保存到其他地方。处理每个通知，因为它到达而不是批量处理。

### 步骤 4：评分、聚合并启动查看器

所有运行完成后：

1. **对每个运行评分** — 启动评分子代理（或内联评分），读取 `agents/grader.md` 并针对输出评估每个断言。将结果保存到每个运行目录的 `grading.json`。grading.json 的期望数组必须使用 `text`、`passed` 和 `evidence` 字段（不是 `name`/`met`/`details` 或其他变体）— 查看器依赖这些确切的字段名。对于可以编程检查的断言，编写并运行脚本而不是目测检查 — 脚本更快、更可靠，并且可以在迭代中重复使用。

2. **聚合成基准** — 从 skill-creator 目录运行聚合脚本：

   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```

   这会生成具有 pass_rate、时间和 tokens 的 `benchmark.json` 和 `benchmark.md`，包括 mean ± stddev 和 delta。如果手动生成 benchmark.json，请参阅 `references/schemas.md` 获取查看器期望的确切 schema。
   将每个带 Skill 版本放在其基线对应版本之前。

3. **做分析师检查** — 读取基准数据并揭示聚合统计可能隐藏的模式。请参阅 `agents/analyzer.md`（"分析基准结果"部分）了解要查找的内容 — 例如总是通过而不考虑 Skill 的断言（无区分力）、高方差 evals（可能不稳定）和时间/tokens 权衡。

4. **启动查看器** — 同时显示定性输出和定量数据：

   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```

   对于迭代 2+，还要传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork/无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 写入独立的 HTML 文件而不是启动服务器。当用户点击"提交所有评论"时，反馈将下载为 `feedback.json` 文件。下载后，将 `feedback.json` 复制到工作区目录中，以便下一次迭代拾取。

   注意：请使用 generate_review.py 创建查看器；无需编写自定义 HTML。

5. **告诉用户** 类似于："我在浏览器中打开了结果。有两个选项卡 — '输出' 让你点击每个测试用例并留下反馈，'基准' 显示定量比较。完成后回到这里告诉我。"

### 用户在查看器中看到的内容

"输出"选项卡一次显示一个测试用例：

- **提示**：给出的任务
- **输出**：Skill 生成的文件，尽可能内联渲染
- **上一个输出**（迭代 2+）：显示上一次迭代输出的折叠部分
- **正式评分**（如果运行了评分）：显示断言通过/失败的折叠部分
- **反馈**：自动保存的文本框
- **上一个反馈**（迭代 2+）：他们上次的评论，显示在文本框下方

"基准"选项卡显示统计摘要：每个配置的平均通过率、时间和 tokens 使用，包括每个 eval 的细分和分析师观察。

导航通过 prev/next 按钮或箭头键。完成后，他们点击"提交所有评论"，将所有反馈保存到 `feedback.json`。

### 步骤 5：阅读反馈

当用户告诉你完成后，读取 `feedback.json`：

```json
{
  "reviews": [
    { "run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..." },
    { "run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..." },
    { "run_id": "eval-2-with_skill", "feedback": "完美，喜欢", "timestamp": "..." }
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。将改进集中在用户有具体投诉的测试用例上。

完成后终止查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进 Skill

这是循环的核心。你已经运行了测试用例，用户已经审查了结果，现在你需要根据他们的反馈使 Skill 变得更好。

### 如何思考改进

1. **从反馈中归纳**。这里发生的大画面是我们试图创建可以运行数百万次（可能 literally，甚至更多）的 Skills，跨许多不同的提示。你和用户在这里迭代的只是几个例子，因为它有助于加快速度。用户非常了解这些例子，他们可以快速评估新输出。但如果你和用户共同开发的 Skill 只适用于那些例子，它就没用了。不要进行棘手的过度拟合的更改，或者压迫性的限制性 MUSTS，如果有一些固执的问题，尝试分支并使用不同的隐喻，或推荐不同的工作模式。尝试相对便宜，也许你会找到很棒的东西。

2. **保持提示精简**。删除没有发挥作用的东西。确保阅读 transcript，而不仅仅是最终输出 — 如果看起来 Skill 让模型浪费了大量时间做没有成效的事情，你可以尝试删除导致它这样做的 Skill 部分，看看会发生什么。

3. **解释为什么**。努力解释你要求模型做的每件事背后的**为什么**。今天的 LLM 很聪明。他们有很好的心智理论，当给出一个好的框架时可以超越死板的指令，真正发挥作用。即使用户的反馈是简短的或沮丧的，也要努力真正理解任务，理解用户为什么写他们写的东西，理解他们实际写了什么，然后将这种理解传递到指令中。如果你发现自己用大写的 ALWAYS 或 NEVER，或者使用超级僵化的结构，那是黄色旗帜 — 如果可能，重新构建并解释推理，让模型理解为什么你要求的事情很重要。这是一种更人性化、更强大、更有效的方法。

4. **寻找跨测试用例的重复工作**。阅读测试运行的 transcript，注意到子代理是否都独立写了类似的辅助脚本或对某事物采取了相同的多步骤方法。如果所有 3 个测试用例都导致子代理编写了 `create_docx.py` 或 `build_chart.py`，这是 Skill 应该捆绑该脚本的强烈信号。写一次，放在 `scripts/` 中，告诉 Skill 使用它。这可以节省每次未来调用重新发明轮子。

这个任务非常重要（我们正在尝试每年创造数十亿美元的经济价值！），你的思考时间不是瓶颈；慢慢来，真正深思熟虑。我建议写一个修订草案，然后用新的眼光看待它并改进。真正尽力进入用户的头脑，理解他们想要什么和需要什么。

### 迭代循环

改进 Skill 后：

1. 将你的改进应用到 Skill
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录中，包括基线运行。如果你在创建新 Skill，基线始终是 `without_skill`（无 Skill）— 这在迭代中保持不变。如果你在改进现有 Skill，根据判断什么作为基线有意义：用户带来的原始版本，还是上一个迭代。
3. 重复评估流程

## 优化 Skill 描述

Skill 描述是主要的触发机制，所以让它们正确很重要！描述应该：

- 准确描述 Skill 做什么
- 具体说明何时使用（用户会说什么/什么上下文）
- 稍微有"推动性"，因为 Claude 有"触发不足"的倾向
