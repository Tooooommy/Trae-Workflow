# 反模式与错误案例专家

记录项目中的错误案例、反模式和失败经验，供后续参考和改进。

## 何时激活

- 发现错误或设计失误时
- 遇到技术债或架构问题
- 项目失败或回滚时
- 代码审查中发现反模式
- 需要总结失败经验

## 核心职责

1. **错误记录** - 记录错误案例和解决方案
2. **反模式沉淀** - 总结常见反模式和避免方法
3. **经验分享** - 沉淀失败经验供团队参考

## 目录结构

```
.trae/rules/
├── README.md
├── bad-practices/
│   ├── YYYY-MM-DD-[错误名称].md
│   └── ...
└── lessons-learned/
    └── YYYY-MM-DD-[经验总结].md
```

## 记录规则

1. **发现时记录** - 发现问题时立即记录
2. **根因分析** - 深入分析根本原因
3. **方案完整** - 提供短期和长期解决方案
4. **可操作** - 确保经验可复用和落地

## 文件命名

```
YYYY-MM-DD-[类型]-[简短描述].md
```

示例：

- `2024-01-15-bug-api-timeout.md`
- `2024-01-20-anti-pattern-singleton-abuse.md`
- `2024-01-25-lesson-database-migration.md`

## 质量门禁

| 检查项 | 要求 |
| ------ | ---- |
| 根因分析 | 必须包含 |
| 解决方案 | 必须包含 |
| 预防措施 | 至少 2 条 |
