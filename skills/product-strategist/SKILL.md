# 产品专家模式

用于产品需求分析与文档编写的技能。

## 何时激活

当用户要求以下任一操作时激活：

- 编写产品需求文档（PRD）
- 编写用户故事（User Story）
- 需求分析与分解
- 定义 MVP（最小可行产品）
- 制定产品路线图
- 需求优先级评估
- 编写产品需求规格说明书

## 核心概念

### 文档类型对照

| 文档     | 用途         | 适用场景           |
| -------- | ------------ | ------------------ |
| PRD      | 产品需求文档 | 功能规划、团队协作 |
| MVP      | 最小可行产品 | 快速验证、敏捷开发 |
| 用户故事 | 需求分解     | 迭代开发、任务拆分 |

### 用户故事 (INVEST原则)

| 原则            | 说明                   | 检查 |
| --------------- | ---------------------- | ---- |
| I - Independent | 独立，不依赖其他故事   | [ ]  |
| N - Negotiable  | 可协商，细节可讨论     | [ ]  |
| V - Valuable    | 有价值，对用户有价值   | [ ]  |
| E - Estimable   | 可估算，能估算工作量   | [ ]  |
| S - Small       | 足够小，一个迭代完成   | [ ]  |
| T - Testable    | 可测试，有明确验收标准 | [ ]  |

**格式**:

```
作为 [角色]
我想要 [功能]
以便 [收益]
```

**验收标准 (Gherkin)**:

```
Given [前置条件]
When [触发动作]
Then [预期结果]
```

### 需求优先级 (MoSCoW)

| 级别   | 说明     | 决策依据             |
| ------ | -------- | -------------------- |
| Must   | 必须有   | 无此功能产品无法上线 |
| Should | 应该有   | 重要但可延迟         |
| Could  | 可以有   | 锦上添花的功能       |
| Won't  | 以后再说 | 本版本不做           |

### 故事点估算

| 点数 | 复杂度 | 工作量  |
| ---- | ------ | ------- |
| 1    | 简单   | < 2小时 |
| 2    | 简单   | 2-4小时 |
| 3    | 中等   | 4-8小时 |
| 5    | 中等   | 1-2天   |
| 8    | 复杂   | 2-3天   |
| 13   | 复杂   | 3-5天   |
| 21   | 需拆分 | > 5天   |

### 需求分解层次

| 层次    | 说明       | 示例       |
| ------- | ---------- | ---------- |
| Epic    | 大型功能集 | 电商系统   |
| Feature | 功能模块   | 订单管理   |
| Story   | 用户故事   | 用户下单   |
| Task    | 开发任务   | 实现购物车 |

## 最佳实践

### 1. 需求分析

- **用户价值导向** - 每个需求都有明确的用户价值
- **可测试** - 需求可转化为可执行的测试用例
- **清晰无歧义** - 避免模糊描述，如"快速"、"友好"
- **独立可拆分** - 需求之间低耦合

## 需求分解技巧

### 从目标倒推

```
最终目标：用户完成订单
    ↓
子目标 1：用户浏览商品
    ↓
子目标 2：用户加入购物车
    ↓
子目标 3：用户结算
```

### 按用户流程分解

```
用户登录 → 用户搜索 → 用户浏览 → 用户下单 → 用户支付 → 订单完成
```

### 按层次分解

| 层次    | 示例                   |
| ------- | ---------------------- |
| Epic    | 完整的电商系统         |
| Feature | 用户下单功能           |
| Story   | 作为用户，我想快速下单 |
| Task    | 实现购物车组件         |
